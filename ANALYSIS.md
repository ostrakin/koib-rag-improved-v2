# Анализ и рефакторинг проекта KOIB RAG

## Что было

Получены два архива одного и того же проекта:

| Архив | Что это | Версия |
|------|---------|--------|
| `Koib_4_12` | Полный проект: индексация **и** обслуживание | 4.11/4.12 |
| `Server_repo` | Урезанная вручную копия «только сервер» | 4.12 |

## Найденные дубли

1. **Весь пакет `src/` (12 модулей) — байт-в-байт скопирован в оба архива.**
   Это главный источник проблем: любую правку приходилось вносить дважды, и
   копии неизбежно расходились бы со временем. Проверка показала, что `src/`
   серверного репозитория полностью идентичен полному.

2. **`_get_pipeline()` — точная копия** в `api/routes/query.py` и
   `api/routes/vk_callback.py`. Каждый файл держал собственный глобальный
   `_pipeline`, из-за чего модель эмбеддингов и FAISS могли загрузиться в
   память **дважды**.

3. **`is_supported_artifact()` / `_artifact_kind()` в `src/artifacts.py`** —
   лишние обёртки над каноническими функциями `src/text_processing.py`.

4. **`src/logging_module.py` — мёртвый код** (не импортируется нигде).

## Что сделано

- **Один источник истины.** Весь код живёт в одном репозитории; ручная копия
  `src/` из `Server_repo` устранена.
- **`_get_pipeline` вынесен** в общий `api/deps.py` (`get_pipeline`,
  `get_vk_service`, `close_pipeline`). Оба роутера и `api/app.py` берут
  синглтон оттуда. Пайплайн теперь гарантированно один на процесс.
- **Обёртки в `artifacts.py` удалены** — используются канонические
  `artifact_kind` / `is_supported_artifact` из `text_processing.py`.
- **Мёртвый `logging_module.py`** перенесён в `extras/query_logger.py`
  (сохранён, но не входит ни в одну сборку).
- **Две точки входа вместо «раздвоенного» `main.py`:**
  `index_app.py` (индексация) и `serve_app.py` (сервер).
- **`export_index.py` заменён** на единый `tools/index_transfer.py`
  (`verify` / `export` / `pack` / `unpack`) — логика проверки/экспорта больше
  не дублируется.

## Разделение модулей (по графу импортов)

| Категория | Модули |
|-----------|--------|
| **Общие** (в обеих сборках) | `config`, `src/{indexing, preprocessing, text_processing, utils}` |
| **Только сервер** | `src/{generation, procedures, quarantine, rag_pipeline, retrieval, safety, validation, vk_bot}`, весь `api/` |
| **Только индексатор** | `src/{parsing, chunking, figure_captioning, artifacts, jsonl_repair, evaluation}`, `batch_ingest.py`, `build_ideal_index.py` |

Тяжёлые зависимости (`pymupdf`, `python-docx`, `pillow`, `pytesseract`) есть
**только** у индексатора. Серверная сборка их не тянет.

## Две сборки

`make_apps.py` собирает из единого исходника две самодостаточные сборки:

```
dist/koib-indexer/   # полный набор для мощного ПК
dist/koib-server/    # узкий набор для сервера (2 ГБ ОЗУ), без OCR/парсинга
```

Общие модули попадают в обе сборки, но **сопровождаются из одного места**
(`src/`), поэтому дублирующегося исходного кода нет.

## Перенос индекса на сервер

1. На индексаторе: `python index_app.py ingest --rebuild`
2. `python index_app.py verify` → `python index_app.py pack --out index_bundle.zip`
3. `pack` кладёт в архив `index_manifest.json` с параметрами сборки (модель
   эмбеддингов, префиксы, размерность, контрольные суммы) и исключает
   рантайм-данные (`metadata/`, `logs/`).
4. Перенести `index_bundle.zip` на сервер (scp/rsync).
5. На сервере: `python -m tools.index_transfer unpack --archive index_bundle.zip`
   — распаковывает индекс и **сверяет манифест с `.env` сервера**: если модель
   или префиксы расходятся, выводит явное предупреждение (это та самая ошибка,
   от которой предостерегал исходный `.env.server.example`).
6. `python serve_app.py --serve`.

## Проверки, которые прошли

- Все `.py` компилируются (синтаксис).
- Серверная сборка не импортирует ни одного «индексаторного» модуля, и эти
  модули физически отсутствуют в `koib-server/src`.
- Все внутренние импорты в обеих сборках разрешаются в существующие файлы.
- Цикл `verify → pack → unpack` отработал на тестовом индексе, включая
  предупреждение при несовпадении модели и защиту от перезаписи без `--force`.
